# Full-Feedback Eval Protocol

## Purpose

The legacy `tests/snapshots/baseline.json` was built from a small cached
feedback set (~700 records, time-split 80/20 → 48 train / 12 test). With
12 test stories, MRR quantized to {0.111, 0.200, 0.250, 0.333} and we
couldn't separate "moderately helpful" features from "slightly helpful"
ones. This new eval uses **all cached feedback** (≥1000 records) and
**k-fold cross-validation** to get a more stable, realistic ranking
quality measurement.

## Data source

- `tests/snapshots/baseline_full.json`
- Generated by `scripts/regen_full_snapshot.py`
- No network calls — reads `.cache/user_feedback/dashboard_feedback.json` only

## Dataset composition

| Field | Count | Source |
|---|---|---|
| Train positives (ups) | ~543 (80% of all ups) | Feedback up records (older 80%) |
| Test positives (ups) | ~135 (20% of all ups) | Feedback up records (newer 20%) — ignored when `--cv` is set |
| Negatives (down + neutral) | ~745 | Feedback non-up records |
| Eval candidates | ~510 | Cached snapshot's HN stories, with neg + train IDs excluded |

Excluded IDs: any story that appears as a `neg_story` or `train_story`
is removed from the candidate pool. This is required by
`RankingEvaluator.evaluate_cv`'s leak-assertion
(`evaluate_quality.py:1017-1019`).

## Eval paradigms

### Drop-one (Run B)

```
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json --cv 5
```

Shows MRR/NDCG/hit/mean_rank for baseline minus each feature.

### Single-feature (Run C — preferred for model quality)

```
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json \
    --cv 5 --single-features --no-drop-one
```

Shows each feature's ability to rank test stories alone. Every feature
except `closest_pos/neg/margin` and `text_len` achieves MRR=1.000 and
hit@30=1.000. Use this to identify which features carry ranking signal.

### How CV works

With `--cv 5`, the harness calls `RankingEvaluator.evaluate_cv()` which
does 5-fold cross-validation over the 678 up-stories (543 train +
135 time-split test, summed). Each fold:

1. Random 80/20 split of all ups → train fold, test fold.
2. Re-compute embeddings for all train+test stories (cached by the
   embedding model).
3. Build candidate pool = `dataset.candidates` (510 snapshot stories)
   ∪ test fold stories not already in candidates.
4. Train a single model on the train fold + neg stories.
5. Rank the candidate pool.
6. Compute MRR, NDCG@30, hit@30, mean_rank, etc. — averaged across folds.

## Interpretation (corrected — critical update)

**The drop-one CV result is a score saturation artifact, NOT a
model-redundancy finding.** Single-feature experiments prove this:

- The 26-feature model gets NDCG@30=0.018 — **25× worse** than
  `story_age` alone (0.448) and **23× worse** than `source_trust`
  alone (0.422).
- The 26-feature model is *strictly worse than almost any single feature
  alone*. Combining all 26 features degrades ranking quality.

**Root cause:** The `OrdinalThresholdModel` in `api/ordinal_model.py:371-376`
clips `at_least_neutral` and `upvote` to `[0,1]` and takes their mean.
With 26 features, the SVM produces confident probabilities → both hit
1.0 for many candidates → massive ties. Tie-breaking determines the
test story's rank, not the model. The single-feature linear model does
not saturate — the score distribution has room to spread.

This means:
- **MRR and NDCG in the drop-one CV are meaningless** — they reflect
  tie-breaking, not feature importance.
- **The single-feature CV is the correct tool** for model quality assessment
  — it avoids saturation by using one feature at a time.
- **Run A (legacy, 48 train) is the correct tool** for feature importance
  — small training set avoids saturation, clear hierarchy emerges.

## Action needed

The model needs either (a) score-domain tie-breaking (e.g., add
`pos_knn` as a small epsilon in the ordinal model to spread ties) or
(b) a reduction to fewer features (5-10) to avoid over-saturating
the SVM.

## When to use which snapshot

| Eval goal | Snapshot | Flag | Why |
|---|---|---|---|
| Fine-grained feature importance | `baseline.json` | (none) | Small dataset avoids saturation |
| Model quality (single-feature) | `baseline_full.json` | `--cv 5 --single-features --no-drop-one` | No saturation, each feature tested alone |
| Quick smoke test | `baseline.json` | (none) | Fast (~3s/eval) |

## Re-generating the snapshot

```sh
uv run scripts/regen_full_snapshot.py
```

This reads `.cache/user_feedback/dashboard_feedback.json` and writes
`tests/snapshots/baseline_full.json`. The new snapshot must be
re-generated whenever the feedback file changes substantially.

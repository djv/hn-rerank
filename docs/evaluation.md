# Evaluation

This file explains what the repo's evaluation tooling measures and how
it's configured. The canonical entrypoint is `scripts/feature_ablation.py`.

## What the eval measures

The evaluator is an offline ranking benchmark, not a live dashboard metric.

Core question: *If a previously-upvoted story appears in the candidate pool,
does the model rank it near the top?*

Default flow:
1. Load a feedback snapshot (`.cache/user_feedback/dashboard_feedback.json`)
2. Build positives from upvotes, negatives from down + neutral
3. Split positives into train vs test by chronological order (oldest → train,
   newest → test)
4. Build a candidate pool from cached HN distractor stories
5. Inject held-out test stories into the candidates so they can be found
6. Train a classifier model from train labels + story features
7. Score and rank all candidates using the trained model
8. Compute ranking metrics against the held-out test stories

What it measures:
- How well the ranking pipeline surfaces held-out stories the user
  previously upvoted

What it does NOT directly measure:
- Live click quality or dashboard feedback quality
- Production refresh latency
- Final display-list behavior (dedup, source diversity) unless `--final-list`
  is used on the legacy evaluator
- Retrieval — held-out test stories are injected into the pool, so the eval
  is a ranking benchmark, not an end-to-end retrieval benchmark

Important caveat: held-out test stories are injected into the candidate pool.
Good metrics mean "if the relevant story is in the pool, the ranking can find
it." They do not prove retrieval discovered it unaided.

## The dataset

The primary snapshot is `tests/snapshots/baseline_full.json`, generated by:

```sh
uv run scripts/regen_full_snapshot.py
```

No network calls — reads `.cache/user_feedback/dashboard_feedback.json` only.
Re-generate whenever the feedback file changes substantially.

### Composition

| Component | Count | Source |
|---|---|---|
| Train stories | ~639 (80% of upvotes) | Feedback up records, older 80% by time |
| Test stories | ~160 (20% of upvotes) | Feedback up records, newer 20% |
| Negatives | ~810 (down + neutral) | Feedback non-up records |
| Eval candidates | ~480 | Cached snapshot's HN stories, with neg + train IDs excluded |
| Feature dims | 16 + 384 raw = 400 (configurable) | 16 metadata features + 384-d title embedding |

Test stories are injected into candidates. Negatives never appear in candidates
— they're pure training signal (excluded by leak-assertion in
`RankingEvaluator.evaluate_cv()`).

An older snapshot `tests/snapshots/baseline.json` (48 train / 12 test) is kept
for fast unit-test evals but should not be used for serious comparisons.

## Two eval protocols

### Time-split (`--cv 0`, default)

Stored 80/20 split by timestamp. Train on older stories, test on newer stories.
Single evaluation. Fast (~5s), deterministic.

Use this for:
- Quick model comparisons during development
- Tuning hyperparameters where speed matters
- Generating head-to-head comparisons (this is what the 4×4 ablation used)

Limitation: single split — metrics can be noisy and may not generalize.

### Cross-validation (`--cv N`)

N-fold cross-validation for stable metric estimates. The full set of
upvotes is divided into N folds; each fold trains on N-1 folds and
tests on the held-out fold. Returns averaged metrics + std-dev across
folds.

Content-based — the eval measures ranking quality from story features
and embeddings, not temporal patterns. All stories are used for
training and testing.

Use this for:
- More stable metric estimates (less noise than single-split)
- Cross-checking time-split findings
- Ablation runs where reliability matters

Limitation: slower (~3× per fold).

## Feature configuration

Features are passed via `--features name=feat1,feat2,...`. Names must match
entries in the `METADATA_FEATURES` or `SIMILARITY_FEATURES` registries in
`api/rerank.py`.

Current 16-feature set (winner from 4×4 ablation):

```
centroid, pos_knn, neg_knn, pos_neg_ratio,
title_len, text_len, is_github, is_pdf,
closest_pos, closest_neg, closest_margin,
is_hn, source_trust, local_density,
cluster_size, embedding_magnitude
```

**`--raw-embedding-features`** prepends 384-dimensional raw title embeddings
to the feature matrix. This gives 16 + 384 = 400 total dimensions.

Features removed from the original 22 (June 2026): `log_points`,
`log_comments`, `comment_ratio`, `comments_count`, `story_age`,
`domain_recency`. The 16-feature set outperforms 22-feature in all model
configs (see 4×4 ablation in AGENTS.md).

## Model configuration

**`--model-type`** sets the classifier. Options (from
`api/config.py:SingleModelConfig`):

| Value | Class | Notes |
|---|---|---|
| `svm` | `SVC(kernel=rbf, C=0.3, gamma=scale)` | Current production default |
| `mlp` | `MLPClassifier(hidden_layer_sizes, activation, alpha, solver, lr_init)` | |
| `random_forest` | `RandomForestClassifier(n_estimators, max_depth, ...)` | |
| `gradient_boosting` | `HistGradientBoostingClassifier` | |
| `logistic` | `LogisticRegression` | |

**`--override key=value`** sets arbitrary `SingleModelConfig` fields. Examples:

```sh
--override svm_c=0.3
--override mlp_activation=logistic
--override mlp_hidden_layers=128,64
--override mlp_solver=lbfgs
--override mlp_alpha=0.1
--override rf_max_depth=10
--override rf_min_samples_leaf=5
```

Override values are type-converted based on the field's annotated type
(int, float, bool, str) in `api/config.py:SingleModelConfig`.

The training config in eval mode relaxes production thresholds:
`min_positive_labels=1`, `min_negative_labels=1`.

## Model architecture

The ordinal threshold model (`api/ordinal_model.py`) trains **two binary
classifiers**:

1. **Neutral model**: predicts P(score >= neutral) — "at least neutral"
2. **Upvote model**: predicts P(score >= upvote) — "upvote"

Both use a `StandardScaler → [classifier]` pipeline. Training labels are
balanced via `_balance_binary_matrix()` when
`SingleModelConfig.balance_training_labels` is True.

Final ranking utility score:

```
utility = clip((P(at_least_neutral) + P(upvote)) / 2.0, 0.0, 1.0)
```

This arithmetic mean of two probabilities is the source of the score
saturation artifact (both probabilities hit 1.0 simultaneously) that
plagued the 26-feature model. The 16-feature model avoids this because
there's enough score spread with fewer (or regularized) features.

## Metrics

Metrics are computed by `evaluate_quality.py:_compute_metrics()`:

| Metric | Definition |
|---|---|
| **MRR** | `1 / (first_relevant_position + 1)`, 0 if none found |
| **NDCG@k** | DCG / ideal DCG with log₂(i+2) discount, binary relevance |
| **Recall@k** | hits / len(relevant_ids) |
| **Precision@k** | hits / k |
| **Hit@k** | 1.0 if any relevant in top-k, else 0.0 |
| **Mean rank** | Average rank of all test stories; `len(ranked_ids) + 1` if none found |
| **Median rank** | Median rank of all test stories |
| **nonhn_score_stddev** | Std-dev of non-HN candidate model scores (plateau detection) |
| **nonhn_at_0.5_fraction** | Fraction of non-HN candidates with score ∈ [0.499, 0.501] |

k defaults to `[10, 20, 30, 50]` in the ablation scripts.

**Primary metric: NDCG@30** — top-30 ranking quality. MRR is informative but
can be 1.000 with poor overall ranking (one good hit, rest buried).

**Plateau detection:** `nonhn_at_0.5_fraction` measures score saturation
— the fraction of non-HN stories tied near the decision boundary. Values
>> 0 indicate the model is producing degenerate scores (too many 1.0s
or 0.5s). A healthy model has `nonhn_at_0.5 == 0.00`.

## The ablation pipeline

`scripts/feature_ablation.py` wraps the evaluator to run multiple
configurations systematically.

### Modes

**Drop-one** (default): runs baseline with all features, then removes
each feature one at a time to measure per-feature importance.

```sh
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json --cv 5
```

**Single-feature**: runs each feature in isolation (requires
`--no-drop-one --single-features`).

```sh
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json \
    --cv 5 --single-features --no-drop-one
```

**Explicit feature sets** (used by 4×4 ablation and tuning scripts): pass
named feature sets via `--features`.

```sh
uv run python scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json \
    --no-drop-one --no-single-features --cv 0 \
    --model-type svm --raw-embedding-features \
    --features 'winners=centroid,pos_knn,neg_knn,...' \
    --override svm_c=0.3
```

### How run_one() works

1. Loads `AppConfig` from `hn_rerank.toml`
2. Overrides `classifier.features` with the passed feature list
3. Optionally sets `raw_embedding_features`
4. Optionally sets `single_model.model_type`
5. Applies arbitrary `--override` key=value pairs
6. Calls `evaluator.evaluate_cv(n_folds, config)` or
   `evaluator.evaluate(config)`
7. Returns an `AblationResult` with all metrics + timing

### Tuning scripts

Two scripts orchestrate larger sweeps by calling `feature_ablation.py`
as a subprocess:

| Script | Cells | Purpose |
|---|---|---|
| `scripts/run_4x4_ablation.sh` | 16 | Feature sets × model types (4 × 4) |
| `scripts/run_svm_tuning.py` | 24 | SVM C × gamma grid + kernel variants |
| `scripts/run_mlp_rf_tuning.py` | 27 | MLP alpha, hidden layers, solver, lr_init + RF |

All results are saved to `results/*.json` as `AblationResult` dicts and
consolidated to `results/4x4_matrix.json`, `results/svm_tuning_results.json`,
`results/mlp_rf_tuning_results.json`.

## Common commands

Full-feedback single-feature CV (model quality assessment):

```sh
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json \
    --cv 5 --single-features --no-drop-one
```

Drop-one CV (feature importance, high noise):

```sh
uv run scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json --cv 5
```

Time-split eval with specific model:

```sh
uv run python scripts/feature_ablation.py tests/snapshots/baseline_full.json \
    --feedback .cache/user_feedback/dashboard_feedback.json \
    --no-drop-one --no-single-features --cv 0 \
    --model-type svm --raw-embedding-features \
    --features 'test=centroid,pos_knn,...' \
    --override svm_c=0.3 --output results/test.json
```

Regenerate the full snapshot:

```sh
uv run scripts/regen_full_snapshot.py
```
